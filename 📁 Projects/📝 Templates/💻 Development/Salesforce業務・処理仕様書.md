## 3. 主フロー（Main Flow）：正常系

> ユーザー操作からシステム内部の自動処理の連鎖・最終状態までを時系列で記載する。

```
[ユーザー操作]
  └─ 商談詳細画面にて「フェーズ = Closed Won」に変更 → 保存

[自動処理の連鎖]
  1. Validation Rule：必須項目チェック（金額・契約開始日）
     ✓ 通過 → 次へ

  2. Before Save Flow（または Trigger）：
     - 商談コードを自動採番・セット

  3. レコード保存（コミット）

  4. After Save Flow（または Trigger）：
     - 関連受注レコード（Order__c）を自動作成
     - 承認申請レコードを作成・承認フロー起動

  5. 外部連携：
     - Slack #sales チャンネルに通知送信（Callout）

[事後状態]
  - 商談.ステータス = "承認待ち"
  - 受注レコードが1件新規作成される
  - Slack に通知メッセージが投稿される
```

| # | エラー発生箇所 | エラー条件 | システム動作 | ユーザーへの表示 | 復旧手順 |

| E-1 | Validation Rule | 金額が未入力 | 保存処理中断・ロールバック | 「金額は必須です」とエラーメッセージを表示 | 金額を入力して再保存 |

## 7. 自動処理マップ（Automation Map）

> このユースケースで発火するすべての自動処理を一覧化する。

| 実行順 | 種別 | 処理名 | トリガー条件 | 処理内容 | 備考 |
|--------|------|--------|------------|---------|------|
| 1 | Validation Rule | 商談クローズ必須項目チェック | フェーズ = Closed Won | 金額・契約開始日の必須チェック | |
| 2 | Before Flow | 商談コード自動採番 | 新規作成 or フェーズ変更時 | 商談コード項目をセット | |
| 3 | After Flow | 受注レコード自動作成 | フェーズ = Closed Won | 受注（Order__c）を作成 | |
| 4 | Approval Process | 商談承認フロー | ステータス = 承認待ち | 上長に承認依頼メール送信 | 金額≥100万円のみ |
| 5 | Apex（非同期） | Slack 通知送信 | After Flow 完了後 | Slack API Callout | Queueable Apex |
