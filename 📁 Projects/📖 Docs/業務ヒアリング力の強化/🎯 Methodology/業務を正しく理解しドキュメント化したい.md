# 業務を正しく理解しドキュメント化したい

システム開発において「業務を正しく理解する」ことは、要件定義の精度・設計の品質・開発後の保守性に直結する。  
本ノートでは、業務理解のために活用される代表的な手法・ドキュメントを整理する。

---

## 業務フロー図（Business Flow Diagram / フローチャート）

### 概要

業務の**手順・条件分岐・担当者（レーン）**を時系列で可視化する図。  
「誰が・何を・どの順番で行うか」を中心に描く。

### 特徴

| 項目 | 内容 |
|------|------|
| 主な目的 | 業務プロセスの手順・フローの整理 |
| 視点 | **人・組織・役割（担当者）** 中心 |
| 時間軸 | 左→右 または 上→下 の線形フロー |
| 表現要素 | 開始/終了、処理、条件分岐（菱形）、レーン |
| 主な記法 | BPMN、UML アクティビティ図、スイムレーン図 |
| ツール例 | draw.io, Lucidchart, Figma, Miro |

### 向いている場面

- 業務の「現状（As-Is）」を整理・可視化したいとき
- 複数部署・役割をまたぐ業務の責任境界を明確にしたいとき
- 業務改善・業務設計（To-Be）の議論のベースを作るとき

### サンプル構造（スイムレーン）

```
顧客        | 注文する → 支払う
社内営業    |           ← 注文受付 → 在庫確認 → ...
倉庫担当    |                                  ← 出荷処理 → ...
```

## イベントストリーミング図（Event Streaming Diagram / イベントストーミング）

### 概要

業務内で発生する**イベント（出来事）**を時系列に並べ、  
それに紐づくコマンド・アクター・リアクション・集約などを可視化する手法。  
**EventStorming**（Alberto Brandolini が考案）が代表的。

### 特徴

| 項目 | 内容 |
|------|------|
| 主な目的 | ドメインイベントの発見・ビジネスロジックの構造把握 |
| 視点 | **イベント（何が起きたか）** 中心 |
| 時間軸 | 左→右 の時系列（発生順） |
| 表現要素 | イベント（橙）、コマンド（青）、アクター（黄）、ポリシー（紫）、集約（黄緑） |
| 主な記法 | EventStorming（付箋ベース） |
| ツール例 | Miro, Mural, EventStorming.com |

### 向いている場面

- DDD（ドメイン駆動設計）の文脈でドメインモデルを発見・設計するとき
- 複雑なビジネスロジックを開発チームと事業担当者が共同で整理するとき
- マイクロサービスの境界（Bounded Context）を議論するとき

### 付箋の色と意味

| 色 | 要素 | 例 |
|----|------|----|
| 🟠 橙 | ドメインイベント（過去形） | 「注文が確定された」 |
| 🔵 青 | コマンド（命令） | 「注文を確定する」 |
| 🟡 黄 | アクター（実行者） | 「顧客」「管理者」 |
| 🟣 紫 | ポリシー／ルール | 「在庫がなければキャンセル」 |
| 🟢 黄緑 | 集約（Aggregate） | 「注文」「商品」 |

## 業務フロー図 vs イベントストリーミング図：比較表

| 比較軸 | 業務フロー図 | イベントストリーミング図 |
|--------|------------|----------------------|
| **中心概念** | プロセス・手順 | イベント（出来事） |
| **主な視点** | 誰が何をするか（役割・担当） | 何が起きたか（状態変化） |
| **適したフェーズ** | 要件定義・業務設計 | ドメイン設計・DDD・マイクロサービス設計 |
| **参加者** | 業務担当者・PM・SE | 開発者・アーキテクト・ドメインエキスパート |
| **ドキュメントの寿命** | 業務が変わるまで長期利用可 | 設計フェーズ中心（探索的） |
| **システム設計との距離** | 遠い（業務寄り） | 近い（実装に直結しやすい） |
| **向いている問題** | 業務の現状把握・改善 | 複雑なドメインの発見・設計 |

> **使い分けの目安**：まず業務フロー図で全体像を把握し、複雑なドメインには EventStorming で深掘りする、という組み合わせが効果的。

## その他の業務理解・ドキュメント化手法

### 1. ユースケース図（Use Case Diagram）

UML の一種。**システムがユーザーに対して提供する機能（ユースケース）**を整理する。  
「誰が・何の目的でシステムを使うか」の概要を示す。

- 適したフェーズ：要件定義初期
- 表現：アクター（人型アイコン）とユースケース（楕円）を線で結ぶ
- 注意：詳細なロジックは表現できないため、フロー図と補完的に使う

---

### 2. シーケンス図（Sequence Diagram）

システム内の**複数のオブジェクト・サービス間のやり取り（メッセージ）**を時系列で可視化。  
API 連携・非同期処理の設計に特に有効。

- 適したフェーズ：詳細設計・API 設計
- 表現：ライフライン（縦線）＋メッセージ（横矢印）
- ツール例：PlantUML、Mermaid、Sequence Diagram（sequencediagram.org）

---

### 3. ER 図（Entity-Relationship Diagram）

**データ構造・エンティティ間の関係**を図示する。  
業務で扱われる「情報」を整理するために用いる。

- 適したフェーズ：DB 設計・データモデリング
- 表現：エンティティ（箱）と関係（線・多重度）
- ツール例：draw.io、dbdiagram.io、MySQL Workbench

---

### 4. ドメインモデル図（Domain Model Diagram）

業務領域の**概念とその関係**を整理するクラス図ベースの図。  
ER 図より「意味・業務ルール」に焦点を当てる。

- 適したフェーズ：DDD の文脈でのモデリング
- 表現：クラス図形式（クラス名・属性・関係性）
- DDD の Aggregate・Value Object・Entity などの概念と対応

---

### 5. ステートマシン図（State Diagram / 状態遷移図）

**オブジェクトや業務ステータスの状態変化**を整理する。  
例：注文のステータス（未確定 → 確定済み → 発送済み → 完了）

- 適したフェーズ：業務ルール・ステータス管理の設計
- 表現：状態（丸）・遷移（矢印）・トリガー（ラベル）
- ツール例：PlantUML、draw.io

---

### 6. コンテキストマップ（Context Map）

DDD における**複数のバウンデッドコンテキスト（境界づけられたコンテキスト）間の関係**を可視化。  
どのチームがどのドメインを扱うか、システム間の依存方向を整理する。

- 適したフェーズ：大規模システムの設計・マイクロサービス分割
- 表現：コンテキスト（ブロック）と依存関係（線・関係ラベル）

---

### 7. ユーザーストーリーマップ（User Story Map）

**ユーザーの行動シナリオ（横軸）× 優先度・リリース計画（縦軸）**を可視化。  
アジャイル開発の要件整理・スプリント計画に有効。

- 適したフェーズ：要件定義・スプリント計画
- 表現：付箋ベース（横：行動フロー、縦：機能の粒度）
- ツール例：Miro、Storiesonboard

---

### 8. ジャーニーマップ（カスタマー / エンプロイージャーニーマップ）

従業員やユーザーの**業務における体験の流れ・感情（不満や要望）・タッチポイント**を可視化する。
業務プロセス図が「手順」を描くのに対し、ジャーニーマップは「現場のペインポイント（負の感情）」に焦点を当てる。

- 適したフェーズ：要件定義初期・業務改善の起点（どこから手を付けるべきかの発見）
- 表現：フェーズ（横軸）× 要素（縦軸：行動・感情の浮き沈み・課題・要望など）

---

## まとめ：手法の選び方

```
業務全体の流れを把握したい
  → 業務フロー図（スイムレーン）

複雑なドメインロジックを発見・整理したい
  → イベントストーミング

システムの機能一覧を整理したい
  → ユースケース図

API・サービス間の通信を設計したい
  → シーケンス図

データ構造を設計したい
  → ER 図 / ドメインモデル図

ステータス管理・状態変化を整理したい
  → ステートマシン図（状態遷移図）

マイクロサービスの境界を設計したい
  → コンテキストマップ + EventStorming

要件整理・スプリント計画をしたい
  → ユーザーストーリーマップ
```
