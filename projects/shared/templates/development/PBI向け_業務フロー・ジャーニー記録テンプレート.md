## 2. 🔄 業務フロー（テキストベースの As-Is / To-Be）

> **目的**: 図を使わずに、業務の「手順・担当者・分岐・システム」を箇条書きの階層構造で明確にする。

### ❌ 現状の業務フロー（As-Is）
1. `[申請者]` 申請書フォーマット（Excel）をダウンロードし、必要事項を記入する。
2. `[申請者]` 申請書を印刷し、捺印する。
3. `[申請者]` 上長のデスクに行き、紙を提出する。
    * **[分岐]** 不備がある場合：`[上長]` が口頭で差し戻し、`1.` に戻る。👉 **(ペイン発生箇所)**
4. `[上長]` 内容を確認し、捺印する。
5. `[上長]` 総務部のボックスに紙を投函する。

### ✨ 改善後の業務フロー（To-Be）
1. `[申請者]` システム（Salesforce）から「新規購入申請」画面を開き入力・保存する。
2. `[システム]` 保存と同時に、自動で `[上長]` に承認依頼（Slack通知 / メール）が実行される。
3. `[上長]` 通知のリンクからシステムを開き、「承認」または「却下（コメント必須）」をクリックする。
    * **[分岐]** 却下された場合：`[システム]` が `[申請者]` に却下通知を送り、`[申請者]` は修正して再申請(`1.`へ)。
4. `[システム]` 承認された場合、ステータスを更新し、`[総務部]` のキューに申請データが入る。

## 3. 📝 PBI要件（機能への落とし込み）

> **目的**: 上記のプロセスを実現するためのシステム要件（機能や設定）を一覧化する。これが実作業のタスク（Sub-task）となる。

* **受入条件（Acceptance Criteria）**
  * [ ] 申請者がSalesforce上で購入申請レコードを作成できること。
  * [ ] 保存時に、直属の上長に対して承認プロセスが自動実行されること。
  * [ ] 承認・却下のアクションが Slack に通知されること。

* **必要な設定・開発要素（メモ）**
  * オブジェクト：`購入申請__c` の作成
  * 項目：`金額`, `理由`, `ステータス` などの追加
  * 自動化：承認プロセス（Approval Process）の設定、Slack 連携フローの作成
