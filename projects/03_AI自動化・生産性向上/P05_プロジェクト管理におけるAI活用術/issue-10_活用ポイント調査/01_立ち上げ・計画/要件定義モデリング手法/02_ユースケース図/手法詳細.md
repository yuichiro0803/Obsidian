# ユースケース図：手法詳細

## 概要
ユースケース図（Use Case Diagram）は、UML（Unified Modeling Language）の一種で、「誰が（アクター）」「何をするか（ユースケース）」を楕円と棒人間で表現する図です。システムとユーザー・外部システムの相互作用を俯瞰的に整理するために使用します。

## 目的と効果
- システムが提供すべき機能の全体像を一覧で把握できる
- ユーザー種別（役割）ごとに使える機能・権限を整理できる
- 外部システムとの連携ポイントが明確になる
- 開発スコープの合意形成に使いやすい（非エンジニアにも直感的）

## 主要な要素
| 要素 | 記号 | 意味 |
| :--- | :--- | :--- |
| **アクター** | 棒人間 | システムを利用する人・外部システム |
| **ユースケース** | 楕円 | システムが提供する機能・動作 |
| **システム境界** | 四角の枠 | システムの内側（開発対象）を区別する境界 |
| **関連線** | 実線 | アクターとユースケースの関係 |
| `<<include>>` | 点線矢印 | 必ず実行される共通処理の呼び出し |
| `<<extend>>` | 点線矢印 | 条件によって追加される拡張処理 |

## 作成ステップ
1. **アクターの特定**：システムを使う人・役割（例：一般ユーザー、管理者、外部API）を洗い出す
2. **ユースケースの洗い出し**：「〜する」という動詞形でシステムの機能を列挙する
3. **関連の整理**：各アクターがどのユースケースを使えるかを関連線で結ぶ
4. **include/extendの整理**：共通処理や条件付き処理を分離する
5. **システム境界の設定**：どこまでが今回の開発対象かを明確にする
6. **レビュー**：ステークホルダーと一緒に「使いたい機能が網羅されているか」を確認

## よくある失敗
- ユースケース名が「データを保存する」など技術的すぎる → 「注文を登録する」など業務目線で書く
- アクターが「ユーザー」だけで役割分けが不十分 → 「営業担当者」「管理者」「倉庫担当者」など具体的に
- ユースケースが細かすぎて「ログイン」「ログアウト」ばかり → 代表的な業務機能に絞る

## チェックリスト
- [ ] 全てのユーザー種別・役割がアクターとして定義されているか
- [ ] 外部システムとのインタフェースがアクターまたはユースケースで表現されているか
- [ ] ユースケース名は業務目線の動詞形になっているか
- [ ] システム境界が明確に引かれているか
- [ ] 全ての主要業務機能がカバーされているか
- [ ] 業務担当者とシステム担当者の両方でレビューしているか
