# 既存システム調査：手法詳細

## 概要
現在稼働しているシステム（レガシーシステム含む）の仕様書、データ構造、ソースコード、利用ログなどを分析し、現行業務の制約や隠れた要件を洗い出す手法。

## 目的と効果
- 現行システムで実現されている業務ロジック（暗黙のルール含む）を明らかにする
- ドキュメント化されていない仕様（属人化した知識）を発掘する
- データ構造や連携先から、移行時に考慮すべき制約や依存関係を特定する
- ユーザーの実際の利用パターンから、本当に使われている機能 / 使われていない機能を見極める

## 準備
- **調査対象システムの特定**: 影響を受けるシステム（本体・周辺・連携先）の一覧化と権限調整。
- **既存ドキュメントの収集**: 設計書、マニュアル、障害履歴等の収集と鮮度確認。
- **調査観点の整理**: 機能一覧、データフロー、性能等、何を知りたいかを定義する。

## 調査項目の枠組み
- **機能面**: 画面・帳票・バッチ一覧、業務ルール、カスタマイズ箇所。
- **データ面**: データモデル（ER図）、データ量・増加傾向、品質（不整合等）。
- **インフラ・非機能面**: アーキテクチャ、性能特性、セキュリティ設定。
- **外部連携**: 連携先一覧、連携方式（API/ファイル等）、データ仕様。
- **運用・保守面**: 運用フロー、既知の課題・技術的負債。

## 調査チェックリストのサンプル
| カテゴリ | 調査項目 | 確認方法 |
| :--- | :--- | :--- |
| **機能** | 画面一覧の作成 | 実機操作 / 設計書 |
| | バッチ・ジョブ一覧の作成 | 運用手順書 / ジョブスケジューラ |
| | 主要な業務ルールの整理 | ソースコード / ヒアリング |
| **データ** | ER図の取得 or 作成 | DB定義 / リバースエンジニアリング |
| | 主要テーブルのレコード数確認 | SQLクエリ |
| **インフラ** | サーバー構成図の取得 | インフラ担当者 / 構成管理ツール |
| | 性能データ（レスポンスタイム等）の確認 | 監視ツール / ログ解析 |
| **連携** | 外部連携先の一覧化 | 設計書 / インフラ担当者へのヒアリング |
| **運用** | 障害履歴の確認 | インシデント管理ツール |
| | 既知の技術的負債のリストアップ | 開発チームへのヒアリング |
