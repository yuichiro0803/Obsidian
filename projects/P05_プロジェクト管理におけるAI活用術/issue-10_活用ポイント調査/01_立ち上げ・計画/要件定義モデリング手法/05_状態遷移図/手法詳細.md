# 状態遷移図：手法詳細

## 概要
状態遷移図（State Transition Diagram）は、あるエンティティ（例：注文、申請、チケット）が「どのような状態を持ち」「どのような条件で状態が変化するか」を図示する手法です。「注文：受付中 → 承認中 → 出荷済み → 完了」のように、業務のワークフロー・ライフサイクルを定義するために使用します。

## 目的と効果
- エンティティが取りうる全ての状態と、その遷移条件を網羅的に定義できる
- 「この状態のとき、どの操作が許可されているか」という業務ルールを可視化できる
- 開発者がステータス管理ロジックを実装する際の仕様書として機能する
- 「こんな状態は存在しないはず」という矛盾を設計段階で発見できる

## 主要な要素
| 要素 | 記号 | 意味 |
| :--- | :--- | :--- |
| **初期状態** | 黒塗り円 | エンティティが生成された直後の状態 |
| **状態** | 角丸四角 | エンティティが取りうる個々の状態 |
| **遷移** | 矢印 | 状態から状態への変化 |
| **イベント** | 矢印上のラベル（例：「承認する」） | 遷移を引き起こす出来事・操作 |
| **ガード条件** | [ ] 内のラベル | イベントが発生する条件（例：[金額 > 30万円]） |
| **終了状態** | 黒塗り円 + 外枠 | エンティティのライフサイクルの終わり |

## 作成ステップ
1. **対象エンティティの特定**：どのエンティティの状態を管理するか定める（例：受注、申請書、チケット）
2. **状態の洗い出し**：エンティティが取りうる全ての状態をリストアップする（未対応、処理中、完了など）
3. **遷移の定義**：各状態から遷移できる先と、そのトリガー（イベント）を整理する
4. **ガード条件の追記**：遷移に条件が必要な場合は [ ] で明記する（例：[承認者が部長の場合]）
5. **操作可否の整理**：各状態での許可操作・禁止操作（例：「完了状態では削除不可」）を対応させる
6. **レビュー**：業務担当者に「このような状態の動き方で合っているか」を確認する

## 状態遷移表との組み合わせ
複雑な遷移を整理する場合は、図と表を併用します。

| From（現在の状態） | イベント | 条件 | To（遷移先の状態） |
| :--- | :--- | :--- | :--- |
| 受付中 | 担当者割り当て | なし | 対応中 |
| 対応中 | 完了報告 | 顧客確認済み | 完了 |
| 対応中 | キャンセル申請 | なし | キャンセル確認中 |
| キャンセル確認中 | キャンセル承認 | なし | キャンセル済み |

## チェックリスト
- [ ] 取りうる全ての状態が定義されているか
- [ ] 初期状態と終了状態が明確か
- [ ] 全ての状態から少なくとも1つの遷移先が存在するか（デッドロック状態の確認）
- [ ] 全ての遷移にトリガー（イベント）が定義されているか
- [ ] ガード条件が必要な遷移に条件が付与されているか
- [ ] 業務担当者のレビューが完了しているか
