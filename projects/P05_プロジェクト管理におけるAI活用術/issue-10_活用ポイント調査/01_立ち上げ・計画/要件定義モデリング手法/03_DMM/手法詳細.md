# DMM（ドメインモデル図）：手法詳細

## 概要
DMM（Domain Model Map / ドメインモデル図）は、業務領域（ドメイン）に登場する概念（エンティティ）とその関係を図示する手法です。ER図よりも概念レベルで表現し、技術的な実装に依存しない業務の本質的な構造を可視化します。要件定義の初期段階で、業務範囲と主要概念の共通認識を形成するために使用します。

## 目的と効果
- 業務領域のキーとなる「もの・こと・役割」を明確にする
- データベース設計（ER図）や機能設計の前提となる業務概念の共通理解を形成する
- ステークホルダーとの合意形成に使いやすい（技術色が薄くビジネス寄りの表現）
- ユースケース図・ER図・フロー図の橋渡し役として機能する

## 主要な要素
| 要素 | 表現 | 意味 |
| :--- | :--- | :--- |
| **エンティティ** | 四角（ボックス） | 業務に登場する概念・もの・人（例：顧客、注文、商品） |
| **関連** | 線 + ラベル | エンティティ間の関係（例：顧客が注文する） |
| **関連の多重度** | 1, *, 0..1 | 1対多、多対多などの数量的な関係 |
| **属性（任意）** | ボックス内のテキスト | エンティティが持つ代表的な特徴 |

## 代表的なエンティティの例（販売業務）
- **顧客** - 注文する主体
- **注文** - 業務トランザクションの中心
- **注文明細** - 注文に含まれる商品ごとの行
- **商品** - 販売対象
- **倉庫** - 在庫の管理場所
- **配送** - 出荷・配達を表す出来事

## 作成ステップ
1. **スコープ確認**：対象とする業務領域を定める（例：受発注業務全体）
2. **エンティティの抽出**：ヒアリング・業務フロー・資料から「名詞」に注目して洗い出す
3. **関連の整理**：エンティティ間の関係を動詞（〜する、〜に属する）で表現する
4. **多重度の追記**：1対1、1対多、多対多などを整理する
5. **レビュー・修正**：業務担当者と一緒に「この図で業務が表現できているか」を確認する
6. **ER図設計へのインプット**：承認後、論理データモデル（ER図）設計に引き渡す

## ER図との違い
| 観点 | DMM | ER図 |
| :--- | :--- | :--- |
| 抽象度 | 高い（概念レベル） | 低い（論理・物理設計レベル） |
| 主な読み手 | 業務担当者、PM | SE、DB設計者 |
| 属性の詳細 | 代表的なものだけ | 全属性・データ型・PK/FKを含む |
| 正規化 | 不要 | 必要 |
| 作成タイミング | 要件定義の初期 | 要件定義の後半〜基本設計 |

## チェックリスト
- [ ] 業務の主要な「もの・こと・役割」が全てエンティティとして含まれているか
- [ ] 関連に主語・述語（動詞）が明確に付与されているか
- [ ] 業務担当者が読んで理解できる表現になっているか（技術用語を排除）
- [ ] スコープ外のエンティティが含まれていないか
- [ ] 業務担当者のレビューが完了しているか
