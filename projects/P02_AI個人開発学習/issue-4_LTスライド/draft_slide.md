## スライド 2｜Antigravity学習手順（2分）

```
AI（Antigravity）を使ったApex基礎学習の進め方

STEP 1：お題を決める
         → 過去の案件・業務で使えそなユースケースを１つ選ぶ
         → 迷ったらAIに「お題を提案して」と相談してOK
         → まずはユースケースを整理することを意識

STEP 2：AIにコードを生成させる
         → 「〇〇というケースで Apex コードを書いて」と指示

STEP 3：AIにコードを解説させる
         → 「このコードの各行を日本語で説明して」と依頼
         → その場で疑問が出たらすぐ聞ける

STEP 4：実際にSalesforce組織に配備して動かしてみる  ← New!
         → Developer Org（無料） or Sandbox に貼り付けて保存
         → 匿名実行（Execute Anonymous）でトリガー・メソッドを動かす
         → エラーが出たらそのエラーをAIに貼り付けて解決してもらう
         → 「見て理解」 → 「実際に動かして体感」で定着する
```

**話すポイント**

まずはAIで学ぶ理由（従来学習との比較）を説明し、その後具体的な手順を解説します。

**【比較】従来学習 vs AI学習**

| 比較軸 | ❌ 従来の教科書学習 | ✅ AI学習 |
|--------|-----------------|---------|
| **学習の入口** | 変数・メソッド・クラスなどの「概念」から始まる | お題（ユースケース）を決めた瞬間に始まる |
| **コードとの距離** | 概念を理解した後でないと書けない | いきなり実案件に近いコードから入れる |
| **実案件イメージ** | ❌ 現場の業務をどう書くかの橋渡しがない | ✅ 業務シナリオそのものでコードが生成される |
| **疑問の解消** | 本を読む→試す→調べる（数時間〜数日） | その場でAIに聞けば即解説（数分） |
| **挫折ポイント** | 「概念はわかったけど何も書けない」 | コードが先にあるため「読める→理解できる」 |

- 教科書の「変数・メソッド・クラスの概念」からの学習だと、**実案件でどう組み合わせるかのイメージが全くつかない**
  - 「じゃあ、商談が更新されたときにどう書けばいいの？」あるいは「どこにDMLを書けばいい？」という実践との間に大きな壁があります
- 一方でAIは最初から「実案件に近いコード」を出すため、「実践から入り、概念をあとで学ぶ」という逆転アプローチが可能になり、理解スピードが圧倒的に上がります

**【手順】AI学習の4ステップ**

- **STEP 1〜3**：お題を決めてAIに投げ、解説させることで「実用例から概念を学ぶ」ループを回します
- **STEP 4**：見て終わらせず、Developer Org等に配備して匿名実行などで動かして体感します。「見て理解」→「動かして定着」のサイクルを作ります

## 付録｜【おすすめ】お題の決め方：フロー → Apex 変換

> 💡 お題の決め方に迷ったら、**既存の Salesforce フローを Apex に変換する**という方法が特におすすめです。

### なぜおすすめ？

- **お題を探す手間がゼロ**：自分がすでに知っているフローがそのままお題になる
- **業務との直結**：実際に動くビジネスロジックを理解しながら学べる
- **比較学習**：フロー（GUIの処理）と Apex（コードの処理）の対応関係がわかる
- **共通のガバナ制限**：フローとApexはどちらもSalesforceの実行基盤（ガバナ制限）を共有しています。フローのロジックをApexに置き換えることで、プラットフォーム共通の制約を意識した実践的なコード設計が身につきます
- **多様なパターンの学習**：フローにおける「非同期ジョブ」や「スケジュール済みジョブ」が、Apexクラスだとどのような記述になるのか、実例を通じて色々なパターンを学ぶことができます
- **画面フローの置き換え**：自動起動フロー以外にも、「画面フロー」を「LWC（フロントエンド） ＋ Apex（バックエンド処理）」に作り替えてみるのもありです。より高度なUI/UXとパフォーマンスを求めるケースの実践トレーニングになります

---

### 手順

```
STEP 1：変換したい Salesforce フローのメタデータ（XML）を取得する

  方法①：VS Code + Salesforce CLI を使う場合
    $ sf project retrieve start --metadata "Flow:フロー名"
    → フロー名.flow-meta.xml がローカルに保存される

  方法②：Setup の「フロー」画面からエクスポート

STEP 2：取得した XML ファイルの内容をコピーして Antigravity に貼り付ける

  プロンプト例：
  「以下は Salesforce フローのメタデータ（XML）です。
  このフローと同等の処理を Apex クラスとして書き直してください。」

STEP 3：生成された Apex コードを解説させる

  「各メソッドの役割を日本語で説明してください」
  「ガバナ制限に注意が必要な箇所はありますか？」
  「このコードをトリガーから呼び出すにはどう書けばいいですか？」
```

### 学習効果

| 観点 | 効果 |
|-----|-----|
| 実務直結 | 日常業務で使っているフローの裏側を理解できる |
| 比較理解 | フローとApexの違い・使い分けがわかる |
| 応用力 | 「フロー限界」を超える複雑なロジックへの入口になる |

## 付録②｜コーディング規約に沿ったApexクラスを作成する方法

> 💡 Antigravityに**コーディング規約をRuleとして登録**することで、常に規約に沿ったコードを生成させることができます。

### 手順

```
STEP 1：Antigravity に Rule（コーディング規約）を追加する

  1. Antigravity の設定画面を開く
  2. 「Rules」または「Memory」に以下の内容を登録する

  登録例：
  ---
  # Apexコーディング規約
  - クラス名はPascalCase（例：AccountTriggerHandler）
  - メソッド名はcamelCase（例：handleBeforeInsert）
  - マジックナンバーは使用禁止。定数クラスに定義すること
  - DMLはtry-catchで囲むこと
  - SOQLはforループ内に記述禁止
  - テストクラスは@isTest アノテーションを付与し、カバレッジ75%以上を目標とする
  ---
  ※ 会社のコーディング規約ドキュメントの内容をそのまま貼り付けてもOK

STEP 2：Apex作成を依頼するプロンプトを投げる

  プロンプト例：
  「以下のユースケースでApexクラスを作成してください。
  登録済みのコーディング規約に従ってください。

  ユースケース：
  商談（Opportunity）が成約（Closed Won）になったら、
  関連する取引先責任者にToDoを自動で作成する」
```

### ポイント

| ポイント | 説明 |
|--------|------|
| Ruleの永続化 | 一度登録すれば、以降のすべての会話に自動で適用される |
| 既存規約の流用 | 社内ドキュメントをそのままコピーして貼り付けられる |
| コードレビュー省力化 | 規約違反のコードが出てくる可能性を最初から排除できる |
| チーム統一 | チーム全員が同じRuleを設定することで、品質を統一できる |
