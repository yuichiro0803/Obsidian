# 03｜Bulk API アーキテクチャ

## 概要
数万〜数千万件におよぶ大規模なデータ（Large Data Volumes: LDV）を効率的に処理するために設計された、非同期型の連携API。

## 主な特徴
- **フォーマット**: CSV, JSON, XMLをサポート（一般的にはデータのオーバーヘッドが少ないCSVが推奨される）。
- **通信方式**: 非同期通信。クライアントはデータを送信後、処理の完了を待たずに別の作業を行い、後から結果（ステータス）を確認する。
- **並列処理**: データを「バッチ」と呼ばれるチャンクに分割し、Salesforceのバックグラウンドで並列処理することで高速な処理を実現。
- **ガバナ制限の緩和**: REST/SOAP APIと比較して、1回のリクエストで処理できるデータ量が桁違いに多く、一括処理に最適化されている。

## ユースケース
- **初期データ移行**: 他システムからSalesforce環境へのマスタデータや過去のトランザクションデータの一括ロード。
- **夜間バッチ処理**: 日次・週次で発生する大量のデータ連携や差分更新。
- **大規模なデータクレンジング・アーカイブ**: Salesforce内の古いデータを一括抽出（バックアップ）し、削除する運用。

## 処理の基本的な流れ（Bulk API 2.0の場合）
1. **ジョブの作成**: 対象オブジェクトと操作（Insert, Update, Upsert, Query等）を指定してジョブを作成。
2. **データのアップロード**: 連携対象データを一括でサーバへアップロード。
3. **ジョブのクローズ**: データのアップロード完了を宣言し、Salesforce側での非同期処理を開始させる。
4. **ステータス確認**: 定期的に処理状況をポーリングし、完了を検知。
5. **結果の取得**: 成功データ、および失敗データ（エラー原因含む）の一覧を取得。

## 設計および運用のポイント
- **排他制御（ロック）への注意**: 並列処理されるため、同じ親レコードに対する子レコードの同時更新や、ルックアップ関係の更新によるレコードロックエラーが発生しやすい。必要に応じて直列処理（Serial）オプションを検討する。
- **ポーリング間隔**: 処理結果の確認（ポーリング）を高頻度で行うとAPIを無駄に消費するため、適切な間隔を設定する。
