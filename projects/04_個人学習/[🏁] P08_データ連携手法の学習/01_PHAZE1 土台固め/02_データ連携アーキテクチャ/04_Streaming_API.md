# 04｜Streaming API アーキテクチャ

## 概要
データやシステムの変更をトリガーとして、ポーリング（定期的な問い合わせ）を行わずに、イベント駆動でリアルタイムなデータ受信・通知を実現するPush型の連携アーキテクチャ。

## 主な特徴
- **通信方式**: 非同期・Push通信。BayeuxプロトコルとCometD（ロングポーリングなどの技術）に基づき、サーバーからクライアントへイベントをプッシュ配信する。
- **リアルタイム性**: レコードの作成や更新といった変更イベントが、ミリ秒から秒単位のわずかな遅延で連携先へ伝達される。
- **API消費の最適化**: 従来の「変更はありますか？」と繰り返しシステムへ尋ねるポーリング方式とは異なり、無駄なAPIコールを大幅に削減できる。

## Salesforceにおける主な実装アプローチ
SalesforceのStreaming APIには、目的に応じていくつかのイベントストリーム（イベントバス）が用意されている。

1. **変更データキャプチャ (Change Data Capture / CDC)**
   - **用途**: オブジェクトのレコード変更（Insert, Update, Delete, Undelete）をまるごと外部へ通知する。
   - **特徴**: 設定が非常に簡単。変更があったフィールドの差分データが自動的に配信されるため、外部データベースとのリアルタイム同期に最適。
2. **プラットフォームイベント (Platform Events)**
   - **用途**: レコードの変更にとらわれない、カスタムのビジネスイベントの送受信（例：「在庫が枯渇した」「注文が確定した」など）。
   - **特徴**: イベントスキーマをカスタムオブジェクトのように独自に定義できる。外部からSalesforceへの通知、Salesforceから外部への通知の双方向で利用可能。
3. **PushTopic / 汎用イベント (従来型)**
   - SOQLクエリに基づいてイベントを定義し、そのクエリの条件に合致するデータ変更があった場合に通知を行う旧世代のアプローチ（現在はCDCやPlatform Eventsの利用が推奨される）。

## ユースケース
- **リアルタイムのデータ同期**: Salesforceの取引先情報が更新されたら、即座に基幹システム上の顧客情報も更新する。
- **UIの即時更新**: コールセンター画面において、システム側で対応ステータスが切り替わった瞬間に、オペレーターの画面をリロードなしで最新状態にする。
- **非同期のプロセス分離**: ある処理の完了をもってイベントを発行し、そのイベントをトリガーに複数の別システム・処理を独立して並行稼働させる（マイクロサービス・イベント駆動アーキテクチャ）。

## 設計および運用のポイント
- **イベントの到達保証と順序性**: メッセージの欠損を防ぐための「リプレイID」を利用した再取得の仕組みや、イベントの配信順序に対する考慮が必要。
- **イベントボリューム**: 短時間に大量のイベントが発生する場合、プラットフォームのイベント配信のガバナ制限（1日あたりの配信件数など）に抵触しないか評価する。
