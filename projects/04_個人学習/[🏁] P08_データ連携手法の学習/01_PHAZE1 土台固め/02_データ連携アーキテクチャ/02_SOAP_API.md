# 02｜SOAP API アーキテクチャ

## 概要
SOAP (Simple Object Access Protocol) は、XMLをベースとした堅牢で厳格なプロトコル。WSDL (Web Services Description Language) によってインターフェースが厳密に定義されるため、エンタープライズシステム間の連携で強みを発揮する。

## 主な特徴
- **フォーマット**: XMLのみを使用。冗長だが型定義が厳格。
- **通信方式**: 同期通信。
- **WSDL（設計図）**: クライアントとサーバー間でやり取りするXMLの構造があらかじめWSDLファイルとして提供され、コンパイル時に型チェックが可能。
- **プロトコル非依存**: HTTP以外（SMTPなど）でもメッセージを転送可能。
- **セキュリティとトランザクション**: WS-Securityなどの標準規格により、高度なセキュリティとトランザクション管理が可能。

## ユースケース
- **レガシー基幹システム（ERPなど）との連携**: 型安全性が強く求められるエンタープライズ領域。
- **複雑なトランザクション**: 複数の操作をまたぐ厳密なトランザクション制御が必要なケース。
- **WSDLからの自動コード生成**: 外部システム側でWSDLを読み込み、JavaやC#のスタブコードを自動生成して開発を効率化したい場合。

## REST APIとの比較と使い分け
| 項目 | SOAP | REST |
| :--- | :--- | :--- |
| **フォーマット** | XMLのみ | 主にJSON (XMLも可) |
| **型定義** | 厳格 (WSDL) | 柔軟 |
| **データサイズ** | 大きい（冗長） | 小さい（軽量） |
| **適用シーン** | 堅牢な基幹連携 | モバイル・Web等の軽量連携 |

## 設計および運用のポイント
- ペイロード（データサイズ）が大きくなりがちなため、帯域幅とパフォーマンスへの影響を考慮する。
- REST APIと同様にSalesforceのAPIリミットを消費する。
