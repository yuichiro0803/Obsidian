# 02｜インタフェース定義書の作成ポイント

データ連携の実装・テストにおいて、最も重要なドキュメントが「インタフェース定義書（IF定義書）」です。
開発者が「これさえあれば実装できる」状態にするための記載ポイントをまとめます。

---

## 1. 共通ヘッダー情報
- **IF ID / 名称**: 管理番号と一目でわかる名称
- **連携方式**: REST API（JSON）, SOAP, Bulk API, CSV連携など
- **方向**: 送信元システム → 送信先システム
- **実行タイミング**: リアルタイム / スケジュール（cron形式など）

## 2. 認証・接続情報
- **エンドポイント**: ベースURL（環境ごとの定義）
- **認証方式**: OAuth 2.0 (Client Credentials / JWT), Basic認証
- **ヘッダー情報**: `Content-Type`, `Authorization` 等の指定

## 3. 項目マッピング（メイン）

| 項目名(和名) | 項目名(英名) | 型 | 長さ | 必須 | Salesforce項目 | 変換・補足ルール |
|:---:|:---:|:---:|:---:|:---:|:---:|:---|
| 取引先名 | AccountName | String | 255 | ◯ | `Account.Name` | なし |
| 業種コード | IndustryCode | String | 2 | ◯ | `Account.Industry` | 変換テーブル(M01)参照 |
| 更新日時 | LastUpdate | DateTime | - | ◯ | `SystemModstamp` | yyyy/MM/dd HH:mm:ss形式 |

### 💡 変換ルールの記載例
- **コード変換**: 送信元の「01」をSalesforceの選択肢「製造業」に変換するマッピング。
- **フォーマット変換**: 日付形式の変換、数値の丸め処理。
- **定数設定**: 全レコードに対し、固定値（例：作成者ID）をセットする。

## 4. リクエスト / レスポンス サンプル
- 実際のJSON/XMLサンプルを記載することで、構造（入れ子構造など）の誤解を防ぎます。

## 5. エラーハンドリング
- **HTTPステータスコード**: 200以外の場合の挙動
- **ビジネスエラー**: 項目バリデーションエラー時のレスポンス形式
- **リトライの可否**: エラー時に送信元で再送すべきか、手動リカバリか

## 6. テストチェックリスト
- 異常系（項目長超過、型不一致、認証NG）のテストケース案
