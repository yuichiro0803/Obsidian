# テスト・移行フェーズ 実践ガイド（Salesforceデータ連携開発）

> **このドキュメントの目的**：ベンダーSE/PLとして、Salesforceデータ連携案件のテスト（結合・総合）〜本番移行フェーズを主導するための「ゴール・成果物・実際の進め方」を整理する。

---

## 📌 このフェーズの全体ゴール

| フェーズ | ゴール | 一言 |
|:---|:---|:---|
| **テスト（結合・総合）** | システム間の疎通と業務シナリオの成立を証明する | 単体では見えない「連携特有のバグ」を潰す |
| **本番移行（リリース）** | 本番環境へ安全にデプロイし、運用を開始する | 失敗した時に「どう戻すか」を明確にしておく |

---

### テストフェーズ

|:---|:---|:---:|

## 🏃 実践的な進め方（ベンダー主導）

### ステップ1：結合テスト環境の依存関係を早期に解決する

連携開発で最も遅延しやすいのが「**相手システムのテスト環境が使えない**」という問題。これを先手で解決する。

| 状況 | ベンダー側の対応（解決策） |
|:---|:---|
| 相手環境が準備完了 | 環境間のネットワーク疎通（ファイアウォール・IPホワイトリスト）のみ早期に確認する |
| 相手環境の完成が遅れている | Salesforce（Apex）側で `HttpCalloutMock` を流用した簡易シミュレーターを作るか、Postman の Mock Server を一時的に利用する |
| 双方が未完成 | IF定義書（リクエスト・レスポンスのJSONフォーマット）のみを絶対の正として、お互いにMockを使用する |

---

### ステップ2：結合テストシナリオは「業務フロー」で作成する

単体の「APIが成功したか」ではなく、「データが作られてから、どう外部に連携され、どう更新されるか」を通してテストする。

```
シナリオの例：【SF】商談成約 → 【外部】請求データ作成 → 【SF】請求ステータス更新
1. Salesforce側で商談を「Closed Won」に変更する
2. 連携バッチ（または非同期処理）が起動し、外部へリクエストが飛ぶ
3. 外部システム側でデータが作成されたことを確認する（Mock応答でも可）
4. 外部システムからSalesforceへ、結果（外部システムのIDなど）がコールバックされる
5. Salesforceの商談の「請求連携ステータス」が「連携済」に変わっていることを確認する
```

---

### ステップ3：本番移行手順を「分刻み」で作成する

リリース当日の混乱を避けるため、移行手順は詳細に作成する。

| 時間帯 | 作業者 | 作業内容 | 備考・注意点 |
|:---|:---:|:---|:---|
| 22:00 | 顧客 | 【業務停止】ユーザーログインの凍結 | プロファイル・権限セットの一時変更 |
| 22:15 | ベンダー | 【デプロイ】変更セット（またはCI/CDツール）で資材を展開 | Apex, Flow, Custom Object 等 |
| 22:30 | ベンダー | 【手動設定】Named Credential の本番パスワード設定 | ※ パスワードは顧客担当者が直接入力する |
| 22:45 | ベンダー | 【手動設定】カスタムメタデータ等の本番URL変更 | |
| 23:00 | 双方 | 【疎通確認】本番環境でのPing（簡易通信テスト） | |
| 23:30 | 顧客 | 【稼働判定】テストデータの送受信による判断 | OKなら業務再開、NGなら切り戻し |

---

### ステップ4：最悪を想定した「切り戻し（ロールバック）計画」の策定

「リリースは必ず成功する」という前提を捨て、「致命的なエラーが起きたらどう戻すか」を顧客と合意しておく。

- **切り戻しの判断基準**: 例「2時間以内に問題が解決できず、翌日の業務開始に影響が出る場合」
- **切り戻しの判断者**: 顧客のプロジェクトオーナー
- **切り戻し手順**:
    - デプロイの取り消し（メタデータの戻し）
    - スケジュール済みジョブ（連携バッチ）の停止
    - ユーザー権限の復旧

---

### ステップ5：フェーズ完了の判断基準（Definition of Done）

- [ ] 結合テスト環境の疎通が確認できている
- [ ] 結合テスト（シナリオテスト）が全件実施され、バグがクローズされている
- [ ] 大量データ連携（Bulk等）のパフォーマンステストが完了している
- [ ] 本番移行手順書が作成され、タイムラインが顧客と合意されている
- [ ] 切り戻し（ロールバック）の基準・手順が顧客と合意されている
- [ ] （リリース後）稼働判定チェックシートの全項目がOKになっている
