## 🏃 実践的な進め方（ベンダー主導）

### ステップ1：詳細設計のスコープを「1IF単位」で分割する

詳細設計は基本設計のIF一覧を起点に、1プログラム = 1設計書の粒度で進める。

```
分割の考え方（例）
├── IF-001：1つのApex Callout クラス
│     ├── handleRequest()：呼び出し制御
│     ├── buildRequestBody()：リクエスト組み立て
│     └── parseResponse()：レスポンス解析
├── IF-002：Batch Apex クラス（start / execute / finish）
└── SFCCommonUtil：全IFで共通の汎用クラス
```

> 💡 **ポイント**: 共通処理（認証・エラー通知・ログ出力）を最初に共通部品として設計しておくことで、後からのIF追加コストが下がる。

---

### ステップ2：エラーハンドリング方針を開発前に統一する

エラー処理の不統一が最もレビュー指摘や工数超過の原因になる。**開発開始前**にチーム全体で以下を合意する。

| エラー種別 | 判断基準の例 | 対応 |
|:---|:---|:---|
| **一時的なエラー** | タイムアウト・503エラー | リトライ（最大3回、指数バックオフ） |
| **恒久的なエラー** | 認証失敗・スキーマ不一致 | 即時中断・エラーログ記録・Slack通知 |
| **部分的な失敗** | バッチ処理中の一部失敗 | 失敗レコードのみ記録し続行（再処理対象とする） |

```apex
// エラーハンドリングの実装例（共通パターン）
try {
    HttpResponse res = http.send(req);
    if (res.getStatusCode() == 200) {
        parseResponse(res.getBody());
    } else if (res.getStatusCode() == 503) {
        // 一時的エラー → リトライキューに積む
        IntegrationErrorLogger.logRetryable(res);
    } else {
        // 恒久的エラー → Slack通知 + ログ記録
        IntegrationErrorLogger.logFatal(res);
    }
} catch (CalloutException e) {
    IntegrationErrorLogger.logFatal(e.getMessage());
}
```

### ステップ3：コーディング規約と命名規則を事前に定義する

複数人で開発する場合、規約の不統一がレビュー工数の増加につながる。

#### Salesforceデータ連携開発向け命名規則の例

| 種別 | 命名規則 | 例 |
|:---|:---|:---|
| Callout クラス | `{システム名}Callout` | `ExternalSystemCallout` |
| Batch クラス | `{対象}BatchSync` | `AccountBatchSync` |
| テストクラス | `{テスト対象クラス}Test` | `ExternalSystemCalloutTest` |
| Custom Metadata | `{略称}_Config__mdt` | `SFInteg_Config__mdt` |
| Named Credential | `NC_{システム名}` | `NC_ExternalSystem` |

### ステップ4：テストクラス設計の注意点

Salesforceでの単体テストは実際のCalloutを実行できないため、**MockのHttpResponse**をセットした状態でテストする。

```
テストクラスの設計方針
├── ① 正常系（200 OK）のレスポンスで処理完了を確認
├── ② 異常系（503 / 401 / 500）でエラーログが記録されることを確認
├── ③ バッチ処理はDatabase.executeBatch()でstart/execute/finishが動くことを確認
└── ④ カバレッジ目標：本番コードの85%以上（Salesforce必須: 75%以上）
```

### ステップ5：コードレビューのチェックポイント

| 観点 | 確認内容 |
|:---|:---|
| **ガバナ制限** | SOQL/DML のfor文内実行がないか。リスト化してバルク処理しているか |
| **Named Credentials使用** | URLやトークンをハードコードしていないか |
| **エラーハンドリング** | catch句で握りつぶし（空のcatch）がないか |
| **テストカバレッジ** | 異常系のテストが含まれているか。75%以上を達成しているか |
| **コメント** | 複雑なロジックに説明があるか。機密情報がコメントに含まれていないか |

### ステップ6：フェーズ完了の判断基準（Definition of Done）

- [ ] 全IFの詳細設計書が作成・レビュー済みである
- [ ] 共通部品（エラーログ・通知・認証）の設計が完了している
- [ ] 全クラスのテストカバレッジが75%以上（目標85%）である
- [ ] 単体テスト記録に全ケースが記載されており、NGが0件である
- [ ] コードレビューのNG指摘がすべてクローズされている
- [ ] Named Credentials・Custom Metadataへの設定値が本番移行手順書に記載されている
