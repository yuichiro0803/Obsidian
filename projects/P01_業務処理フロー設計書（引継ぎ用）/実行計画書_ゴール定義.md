---

## 1. なぜこのプロジェクトをやるのか（Why）

### 現状の問題

```
業務フロー図と自動処理の仕様が別々の場所に分散している

具体的な弊害：
  ・業務の手順（誰が何をするか）とシステムの挙動が結びつかない
  ・引き継ぎ先が全体像を把握するために複数ドキュメントを横断する必要がある
  ・「なぜこの仕様にしたか」の根拠が残らない
```

### このプロジェクトで解決すること

> **業務フローとシステム動作を一つの設計書に統合し、引き継ぎ・保守を効率化する**

---

## 2. 最終ゴール（What）

### ゴール文（1文で言うと）

> **局地的な業務処理フローを「ID連結型統合設計書」として一元管理できる体制が整うこと**

### ゴール状態の定義

| # | ゴール | 達成基準 |
|---|-------|---------|
| G1 | 業務フロー（手動操作）と自動処理を一つの図で表現できる | シーケンス図・状態遷移図・処理インデックスの3点セットが揃っている |
| G2 | 処理IDで各図が連結され、外部設計書へ参照できる | `[P-XX]` でシーケンス図・状態遷移図・テーブルが一致している |
| G3 | チームが参照・更新できる形で管理されている | GitHubにプッシュされ、引き継ぎ先がすぐ開ける状態 |
| G4 | Salesforceにおける複数オブジェクト・複数項目の状態遷移を記述できる | パターン集（GUIDE）を使って実際の業務フローに適用できる |

---

## 3. スコープとスケジュール（Scope / When）

### 対象範囲

| 対象 | 内容 |
|------|------|
| **対象業務** | Salesforce導入支援案件の主要業務フロー |
| **対象ドキュメント** | 統合設計書テンプレート・Salesforce記述ガイド・サンプル設計書 |
| **保管場所** | GitHub（Obsidianリポジトリ）/ `projects/P01_業務処理フロー設計書（引継ぎ用）/` |

### スコープ外

- 個別案件の設計書の完成（テンプレートと手法の整備が本プロジェクトのスコープ）
- Salesforceの実装・開発作業

### フェーズ別スケジュール

```
Phase 1｜基盤整備（完了）
  ├─ Q000：業務フロー統合方針の策定
  ├─ Q001：業務仕様書の課題と解決アプローチ
  └─ 統合設計書テンプレート・Salesforceガイドの作成

Phase 2｜実案件への適用（〜Q1末）
  └─ 既存の主要業務フローに本テンプレートを適用する

Phase 3｜標準化・定着（Q2以降）
  └─ チームで参照・更新できるルーティンを確立する
```
