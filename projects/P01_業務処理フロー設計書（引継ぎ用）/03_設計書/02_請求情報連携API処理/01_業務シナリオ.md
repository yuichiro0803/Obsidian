# 01｜業務シナリオ（ユースケース）

> [🔙 INDEX](./00_INDEX.md)

## 1. ユースケース記述

- **ユースケース名**: ERP連携処理（請求先マスタおよび請求書情報）
- **アクター**: 
  - 連携元システム（SmartDB）
  - 外部連携基盤・ERP（ミドルウェア等）
- **事前条件**:
  - 対象となる請求先（BillingInformation__c）または請求書情報（InvoiceHeader__c）が存在すること
  - 各APIエンドポイントへのアクセス権限（認証）が連携元システムに付与されていること

---

## 2. メインフロー

本プロセスは「請求先（BillingInformation__c）」および「請求書（InvoiceHeader__c）」の双方に対して、以下の3つのフェーズで非同期に実行されます。

### フェーズ 1｜ERP連携基盤からの対象データ抽出（抽出API起電）
1. 定期実行スケジューラ等のERP側基盤から、Salesforceのデータ取得API（GET）へリクエストが送信される。
   - 請求先取得: `/BillingInfoAcquisition/*`
   - 請求書取得: `/BillingDataAcquisition/*`
2. Salesforce側で、ステータスが「確定済（未連携）」のレコードを最大件数（LIMIT）まで取得する。
3. 取得したレコードデータを、指定された並び順でCSV文字列として生成し、HTTPレスポンス（`text/csv`）として返却する。
   - **請求書取得時の特殊ロジック**: 同一請求書（ReferenceNo）内の明細が複数のCSVに跨らないよう、LIMIT境界で参照番号が一致するレコードは次回の抽出回に回される。
4. 抽出対象となったレコードの対象ステータス（SendStatus__c または Status__c）を、「連携中」に更新する。

### フェーズ 2｜ERPからの連携結果（完了フラグ）更新
1. ERP側でのデータ取り込み処理完了後、Salesforceの結果更新API（POST）へリクエストが送信される。
   - 請求先連携結果: `/BillingInfoApprove/*` 
   - 請求書連携結果: `/CompletionFlagUpdate/*`
2. Salesforce側でJSONを受信し、対象レコードを検索する。
3. 連携ステータスが「成功 (1)」の場合はステータスを「連携済」に更新し、各種ドキュメントIDを維持する。
4. 連携ステータスが「失敗 (2)」の場合はステータスを「連携エラー」に更新し、エラー詳細（ErrorDetail__c）を記録する。

### フェーズ 3｜SmartDBからの承認ステータス・Webhook受信
1. 連携元（SmartDB）から承認フローのステータス変化に伴い、Salesforceのステータス更新API（POST）へWebhookリクエストが送信される。
   - 請求先向け: `/BillingInfoStatusUpdate/*`
   - 請求書向け: `/InvoiceHeaderStatusUpdate/*`
2. Salesforce側でJSONペイロードを受信・解析し、対象レコード（請求先・請求書）を検索する。

#### ■ 請求先（BillingInformation__c）のステータスマッピング

## 3. 事後条件

- **【フェーズ 1】**
  - 抽出された対象レコードのステータスが「連携中」となっていること
  - ERP連携基盤へCSVフォーマットでデータが引き渡されていること
- **【フェーズ 2】**
  - 対象レコードの最終ステータスが「連携済」または「連携エラー」となっていること
  - エラー発生時は、エラー詳細内容がレコードに保持されていること
- **【フェーズ 3】**
  - 対象レコードへSmartDB側の承認・処理状況が正しく同期されていること
  - 更新失敗時にはエラーメールが指定アドレスへ送信されていること

---

## 4. 例外ケース

- **【共通】不正なリクエスト・対象データ不在**
  - 送信されたJSONの形式が想定外、または対象となる「参照番号（Name）」が存在しない場合、`400 Bad Request` もしくは `404 Not Found` のエラーステータスで応答を返す。
  - レコード更新自体はロールバックされ状態は維持される。
  - 管理者へエラー詳細のメール通知が自動で行われる。
- **【フェーズ 3】Webhook実行日時の逆転**
  - リクエストに含まれる `updated_at` (実行時間) が Salesforce側にすでに保存された時間より古い場合、順序異常と見なして `ExecutionTimeException`（500エラー）をスローし、データを更新しない。
