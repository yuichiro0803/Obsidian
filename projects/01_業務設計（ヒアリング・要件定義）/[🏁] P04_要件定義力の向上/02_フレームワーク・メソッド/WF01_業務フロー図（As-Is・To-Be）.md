# WF01｜業務フロー図（As-Is / To-Be）

## 概要
ウォーターフォールの要件定義工程において、現行業務（As-Is）と理想の業務（To-Be）を視覚化する最も基本的な手法。

## 目的
- 現行業務の課題・非効率を関係者全員で共有する。
- システム導入後の業務の変化・改善効果を「絵」で合意する。

## 図の構成要素
| 要素 | 説明 |
| :--- | :--- |
| **スイムレーン** | 担当部門・職制ごとの実行レーン |
| **アクティビティ** | 各部門が行う作業（四角で表現） |
| **判断分岐** | 条件に応じた処理の分岐（ひし形） |
| **成果物** | 書類・データ・承認など業務の産出物 |
| **システム境界** | 手動作業かシステム処理かの境界線 |

## 作成ステップ
1. **ヒアリング段階**: 現行業務を担当者から聞き出し、ラフなAs-Isを手書きで作成。
2. **確認・修正**: 作成したフローを担当者に見せ、漏れ・誤りを修正。
3. **To-Be設計**: システム導入後の理想フローを設計し、As-Isとの差分（システム化部分）を明確化。
4. **合意形成**: ステークホルダーとのWalkthrough（ウォークスルー）でフローを承認。

## Salesforce実装との連携
- To-Beフローの「自動化部分」が、フロー（Flow）の設計につながる。
- 「システム境界」を明確にすることで、外部連携（API）の必要性が浮き彫りになる。
