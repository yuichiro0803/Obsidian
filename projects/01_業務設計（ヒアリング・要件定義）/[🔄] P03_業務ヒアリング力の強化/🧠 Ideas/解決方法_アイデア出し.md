# 🧠 解決方法：アイデア出しドキュメント

## 1. 現状の課題と原因の分析 (Problem & Root Cause)

### 根本原因 (Root Cause)
**C4「完了イメージ」の欠如**：「どのような情報が揃えばヒアリング終了と言えるか（ゴール）」が具体的に定義されていないため、以下の課題が生じている。

### 派生する課題 (Derived Problems)
| ID | 課題 (事象：何が起きているか) | 原因 (背景：なぜ起きているか) |
| :--- | :--- | :--- |
| **C1** | **ヒアリングのバラツキ** | メンバー間の経験値の差により、確認すべき深さや範囲の基準が共有されていない |
| **C2** | **設計プロセスの属人化** | ユーザーストーリーから具体的な質問案を導き出すための「思考の型」が言語化されていない |
| **C3** | **成果物の品質不均一** | ヒアリング結果を業務フローや仕様書に落とし込む際の精度に、客観的な基準がない |

## 2. 解決アプローチのマッピング (Solution Mapping)
| 解決すべき原因 | 解決アプローチ (Approach) | 期待される効果 |
| :--- | :--- | :--- |
| **C2, C4** | **[💡F] 質問の観点シート** | **ヒアリング前に「何を聞くべきか」の切り口を整理・共有し、思考の漏れを防ぐ** |
| **C1, C4** | **[💡E] ヒアリング完了定義 (DoH)** | **ゴール（必須確認項目）を明文化し、メンバーが過不足なくヒアリングを完遂できるようにする** |
| **C1, C3** | **[💡D] バックログテンプレート** | **典型的なPBIに対する必須ヒアリング項目や成果物イメージを型化し、迷いをなくす** |

## 3. 各アプローチの詳細 (Measures Detail)

### [💡F] 質問の観点シート (Question Perspectives：準備・設計用)
- **内容**: 業務領域（データ連携、バッチ、UI等）ごとに、「アクター、タイミング、例外、制限事項」といった **確認すべき多角的な視点 (Perspective)** をリスト化し、事前に共有・利用する。
- **メリット**: 経験の浅いメンバーでも、プロと同じ「切り口」でヒアリングを設計可能になる。

### [💡E] ヒアリング完了定義 (DoH: Definition of Hearing：完了判定用)
- **内容**: 「このユーザーストーリーなら、アクター・入出力・例外・Salesforce標準制限の4点は必ず明記する」といった、次の工程（設計・開発）へ進むための **最低合格基準** を策定する。
- **メリット**: 「完了イメージ」が具体化されるため、手戻りの防止とメンバーの安心感につながる。

### [💡D] 成功事例のバックログテンプレート (Examples：参照用)
- **内容**: 典型的なPBI（例：CSV出力、API連携）ごとに、「理想のヒアリング項目・成果物イメージ」をセットにしたテンプレート。
- **メリット**: ゼロから考える必要がなくなり、作業のスピードと品質を両立できる。

## 4. 実行優先度とアクションプラン
1. **【準備】観点の整理:** [💡F] 質問の観点シート（基本セット）を作成し、メンバーに共有する。
2. **【基盤】ゴール定義:** [💡E] ヒアリング完了定義 (DoH) の共通ルールを策定する。
3. **【型化】テンプレート:** [💡D] 典型的なPBIに対するバックログテンプレートの第一弾（例：データ連携）を作成する。
