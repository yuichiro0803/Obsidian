# 🧠 解決方法：アイデア出しドキュメント

## 1. 現状の課題と根本原因 (Problem & Root Cause)

### 根本原因 (Root Cause)
**C4「完了イメージ」の欠如**：「どのような情報が揃えばヒアリング終了と言えるか（ゴール）」が具体的に定義されていないため、以下の課題が生じている。

### 派生する課題 (Derived Problems)
| ID | 課題 (事象：何が起きているか) | 原因 (背景：なぜ起きているか) |
| :--- | :--- | :--- |
| **C1** | **ヒアリングのバラツキ** | メンバー間の経験値の差により、確認すべき深さや範囲の基準が共有されていない |
| **C2** | **設計プロセスの属人化** | ユーザーストーリーから具体的な質問案を導き出すための「思考の型」が言語化されていない |
| **C3** | **成果物の品質不均一** | ヒアリング結果を業務フローや仕様書に落とし込む際の精度に、客観的な基準がない |

## 2. 解決アプローチのマッピング (Solution Mapping)
| 解決すべき原因 | 解決アプローチ (Approach) | Backlogツールでの運用イメージ |
| :--- | :--- | :--- |
| **C2, C4** | **[💡F] 質問の観点シート** | ヒアリング前の「下書き」として、Wikiや個人メモで思考を整理・共有するためのガイドとして利用する |
| **C1, C4, C3** | **[💡D] 構造化PBIテンプレート** | **Backlogの「課題詳細」に直接貼り付けて更新する、DoH(完了定義)連動型の入力フォーマット** |

## 3. 各アプローチの詳細 (Measures Detail)

### [[01_施策詳細/02_構造化PBIテンプレート.md|[💡D] 構造化PBIテンプレート (Structured PBI Template)]]
- **内容**: 議事録とバックログテンプレートの役割を統合した、Backlog課題（PBI）専用のMarkdownフォーマット。アクター、入出力項目、正常系・例外系ロジック、Salesforce標準制限など、「埋めるべき枠」があらかじめ定義されている。
- **運用**: ヒアリング中にこのテンプレートに従ってBacklogの課題詳細を直接更新（またはメモ）していく。
- **メリット**: **情報の二重管理（議事録→バックログ転記）を排除**できる。また、全項目が埋まることで自然と「ヒアリング完了(DoH)」の状態が保証される。

### [[01_施策詳細/01_質問の観点シート.md|[💡F] 質問의 観点シート (Question Perspectives)]]
- **内容**: 業務領域ごとに「チェックすべき多角的な視点」をリスト化した設計ガイド。PBIテンプレートを埋めるために「何を聞くべきか」という思考をサポートする。

## 4. 実行優先度とアクションプラン
1. **【標準化】項目定義**: どのような項目があれば「ヒアリング完了」と言えるか（DoH）を、最終成果物から逆算して再定義する。
2. **【武器】テンプレート作成**: 定義した項目をBacklogの課題詳細にそのまま貼り付けられる「Markdownテンプレート」として作成する。
3. **【準備】観点の整理**: テンプレートを埋めるための思考を助ける「質問の観点シート」を整備・共有する。
