# 🧠 解決方法：アイデア出しドキュメント

## 1. 現状の課題と原因の分析 (Problem & Root Cause)
| ID | 課題 (事象：何が起きているか) | 原因 (背景：なぜ起きているか) |
| :--- | :--- | :--- |
| **C1** | **ヒアリングのバラツキ** | メンバー間の経験値の差により、確認すべき深さや範囲の基準が共有されていない |
| **C2** | **設計プロセスの属人化** | ユーザーストーリーから具体的な質問案を導き出すための「思考の型」が言語化されていない |
| **C3** | **成果物の品質不均一** | ヒアリング結果を業務フローや仕様書に落とし込む際の精度に、客観的な基準がない |
| **C4** | **「完了イメージ」の欠如** | **「どのような情報が揃えばヒアリング終了と言えるか（ゴール）」が具体的に定義されていない** |

## 2. 解決アプローチのマッピング (Solution Mapping)
| 解決すべき原因 | 解決アプローチ (Approach) | 期待される効果 |
| :--- | :--- | :--- |
| **C1, C4** | **[💡E] ヒアリング完了定義 (DoH)** | ゴール（必須確認項目）を明文化し、メンバーが自信を持ってヒアリングを完遂できる状態にする |
| **C2, C4** | **[💡B] 自動チェックリスト** | ユーザーストーリーから最適な質問案をAIが即座に提示し、検討の「抜け漏れ」を物理的に防ぐ |
| **C1, C3** | **[💡A] AIシミュレーター** | 模擬実演を通じ、想定外の質問への対応力や、情報の「粒度」への気づきを与える |
| **C2, C3** | **[💡C] 相互レビュー** | 定義(DoH)に基づき、必要な情報が十分に揃っているかを他者が客観的に担保する |
| **C1, C3** | **[💡D] パターン・ライブラリ** | 典型的な業務の「理想の最終成果物（フロー図等）」を型化し、目指すべきゴールを視覚化する |

## 3. 各アプローチの詳細 (Measures Detail)

### [💡E] ヒアリング完了定義 (DoH: Definition of Hearing)
- **内容**: 「このユーザーストーリーなら、アクター・入出力・例外・Salesforce標準制限の4点は必ず埋める」といった完了基準を策定する。
- **メリット**: 「完了イメージ」が共有されるため、迷いがなくなる。

### [💡A] AIヒアリング・シミュレーター
- **内容**: AIを顧客担当者に見立てて、ヒアリングの練習とフィードバックを行う。

### [💡B] ユーザーストーリー連動型チェックリスト自動生成
- **内容**: PBIを入力すると、DoHに基づいた具体的な質問リストをAIが生成する。

### [💡C] 相互レビューの儀式化（ピア・ラーニング）
- **内容**: ヒアリング前の設計内容や、ヒアリング後の「DoH充足度」を相互にチェックする。

### [💡D] 成功事例のパターン・ライブラリ
- **内容**: 業務領域（例：売上連携）ごとの「理想の最終イメージ」をカタログ化し、ゴールとしての参照先を作る。

## 4. 実行優先度とアクションプラン
1. **【最優先】基盤:** [💡E] ヒアリング完了定義 (DoH) の基本ルールを策定する。
2. **【補助】効率:** [💡B] ユーザーストーリーからDoHを埋めるためのプロンプトを試作する。
3. **【定着】運用:** [💡C] 策定した定義に基づき、短時間の相互レビューを試験導入する。
