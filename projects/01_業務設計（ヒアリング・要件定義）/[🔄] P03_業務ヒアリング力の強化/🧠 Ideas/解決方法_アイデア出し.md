# 🧠 解決方法：アイデア出しドキュメント

## 1. 現状の課題と原因の分析 (Problem & Root Cause)

### 根本原因 (Root Cause)
**C4「完了イメージ」の欠如**：「どのような情報が揃えばヒアリング終了と言えるか（ゴール）」が具体的に定義されていないため、以下の課題が生じている。

### 派生する課題 (Derived Problems)
| ID | 課題 (事象：何が起きているか) | 原因 (背景：なぜ起きているか) |
| :--- | :--- | :--- |
| **C1** | **ヒアリングのバラツキ** | メンバー間の経験値の差により、確認すべき深さや範囲の基準が共有されていない |
| **C2** | **設計プロセスの属人化** | ユーザーストーリーから具体的な質問案を導き出すための「思考の型」が言語化されていない |
| **C3** | **成果物の品質不均一** | ヒアリング結果を業務フローや仕様書に落とし込む際の精度に、客観的な基準がない |

## 2. 解決アプローチのマッピング (Solution Mapping)
| 解決すべき原因 | 解決アプローチ (Approach) | 期待される効果 |
| :--- | :--- | :--- |
| **C2, C4** | **[💡F] 質問の観点シート** | **ヒアリング前に「何を聞くべきか」の切り口を整理・共有し、思考の漏れを防ぐ** |
| **C1, C4** | **[💡E] ヒアリング完了定義 (DoH)** | ゴール（必須確認項目）を明文化し、メンバーが自信を持ってヒアリングを完遂できるようにする |
| **C2, C4** | **[💡B] 自動チェックリスト** | ユーザーストーリーから最適な質問案をAIが即座に提示し、検討の「抜け漏れ」を物理的に防ぐ |
| **C2, C3** | **[💡C] 相互レビュー** | 定義(DoH, 観点)に基づき、準備や結果が十分であるかを他者が客観的に担保する |
| **C1, C3** | **[💡D] バックログテンプレート** | **典型的なPBIに対する必須ヒアリング項目や成果物イメージを型化し、迷いをなくす** |

## 3. 各アプローチの詳細 (Measures Detail)

### [💡F] 質問の観点シート (Question Perspectives)
- **内容**: 業務領域（データ連携、バッチ、UI等）ごとに、「アクター、タイミング、例外、制限事項」といった **確認すべき多角的な視点 (Perspective)** をリスト化し、事前に共有する。
- **メリット**: ゼロから考える負担を減らし、メンバーが自律的に「質の高い質問」を設計できるようになる。

### [💡E] ヒアリング完了定義 (DoH: Definition of Hearing)
- **内容**: 「このユーザーストーリーなら、アクター・入出力・例外・Salesforce標準制限の4点は必ず埋める」といった完了基準を策定する（アウトプットの定義）。


### [💡B] ユーザーストーリー連動型チェックリスト自動生成
- **内容**: PBIを入力すると、DoHや質問の観点に基づいた具体的な質問案をAIが生成する。

### [💡C] 相互レビューの儀式化（ピア・ラーニング）
- **内容**: ヒアリング前の観点整理や、ヒアリング後の「DoH充足度」を相互にチェックする。

### [💡D] 成功事例のバックログテンプレート
- **内容**: 典型的な構成（例：CSV出力、API連携）を伴うバックログアイテム（PBI）ごとの「理想のヒアリング項目・成果物イメージ」をテンプレート化する。

## 4. 実行優先度とアクションプラン
1. **【準備】観点の整理:** [💡F] 質問の観点シート（基本セット）を作成し、メンバーに共有する。
2. **【基盤】ゴール定義:** [💡E] ヒアリング完了定義 (DoH) の基本ルールを策定する。
3. **【効率】AI活用:** [💡B] ユーザーストーリーからDoH/観点を埋めるためのプロンプトを試作する。
4. **【定着】運用:** [💡C] 策定したシートに基づき、短時間の相互レビューを導入する。
