# 施策1｜ユーザーストーリーからヒアリングを設計する

↑ 施策一覧：[[README]]

> **目的**：PBI に書かれているユーザーストーリーを読み解き、「何を・誰に・どう聞くか」を自分で設計できるようになる。

---

## なぜユーザーストーリーからヒアリングを設計するのか？

スクラム開発の PBI には、以下のようなユーザーストーリーが書かれている。

```
As a [ユーザーの種類]
I want to [実現したいこと]
So that [得られる価値・理由]
```

このユーザーストーリーは「ユーザーが何を望んでいるか」の出発点に過ぎない。  
実際にシステムを正しく作るためには、その背景にある **業務の手順・例外・感情** まで理解する必要がある。

---

## ユーザーストーリーの読み解き方

### Step 1：「As a」から**ヒアリング対象者**を特定する

| ユーザーストーリーの例 | ヒアリング対象 |
|----------------------|--------------|
| `As a 営業担当者` | 実際に営業業務をやっている現場メンバー |
| `As a 承認者（上長）` | 承認フローをこなしている管理職 |
| `As a 管理者` | システム設定・マスタ管理担当者 |

> 💡 **コツ**：「As a」の人物が複数いる場合、それぞれにヒアリングが必要なことが多い。

---

### Step 2：「I want to」から**現在の業務のやり方（As-Is）**を想像する

「I want to [自動化してほしい]」という要望の裏には、**今は手作業でやっている業務**が必ずある。

#### ヒアリング前の仮説立て

| ユーザーストーリー | 想定 As-Is（仮説） |
|------------------|------------------|
| `日報を自動集計してほしい` | 毎日手でExcelに転記している可能性 |
| `承認を素早く完了したい` | 紙やメールでの承認フローが遅い可能性 |
| `在庫を一元管理したい` | 複数システム・Excelで管理が分散している可能性 |

> 💡 **コツ**：「なぜ自動化/改善したいのか？」という質問で、現在の痛みを引き出せる。

---

### Step 3：「So that」から**ヒアリングの観点（ゴール）**を定める

「So that」が示す価値は、ヒアリングで確認すべき**最終的な成果指標**になる。

| So that（価値） | ヒアリングで確認すること |
|----------------|----------------------|
| `時間を節約したい` | 現在の作業にかかる時間・頻度 |
| `ミスを減らしたい` | どんなミスが・どのくらいの頻度で起きているか |
| `情報を共有したい` | 現在の共有方法・誰が見ているか・何が漏れているか |

---

## ヒアリング設計シート（テンプレート）

```markdown
## ヒアリング設計

**対象PBI（ユーザーストーリー）**
As a ___________
I want to ___________
So that ___________

---

**ヒアリング対象者**：

**ヒアリングゴール**（何を明らかにすれば設計できるか）：

**事前仮説**（As-Is の想像）：
- 

**質問リスト**：
1. （現状の業務フローを把握する質問）
2. （ペインポイントを引き出す質問）
3. （例外ケースを確認する質問）
4. （期待する状態＝To-Beを確認する質問）

**注意点 / 事前確認事項**：
- 
```

---

## よくある質問とコツ

| シーン | 良い質問の例 |
|--------|------------|
| As-Is を把握したい | 「今はどのような手順でやっていますか？」 |
| 課題を引き出したい | 「その中で一番手間がかかる・面倒なのはどこですか？」 |
| 例外ケースを確認 | 「うまくいかないときはどう対処しますか？」 |
| To-Be をすり合わせ | 「理想的にはどういう状態になれば一番助かりますか？」 |
| 優先度を確認 | 「一番改善したい部分はどこですか？」 |
