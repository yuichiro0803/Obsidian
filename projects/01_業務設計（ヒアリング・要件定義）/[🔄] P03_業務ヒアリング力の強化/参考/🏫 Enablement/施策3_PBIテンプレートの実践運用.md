# 施策3｜PBIテンプレートの実践運用

↑ 施策一覧：[[README]]

> **目的**：ヒアリング結果を、PBI に構造的かつ継続的に残せるチーム習慣を作る。

---

## なぜPBIテンプレートに残すのか？

| 記録しない場合 | テンプレートで記録した場合 |
|--------------|----------------------|
| ヒアリング内容がPLの頭の中にだけある | チーム全員が経緯・背景を参照できる |
| 後から「なぜこの仕様にしたか」が分からなくなる | 設計の根拠が残る |
| 新しいメンバーが業務を理解するのに時間がかかる | ドキュメントでキャッチアップできる |

---

## PBIテンプレートの構成

全体構成の詳細は → [[PBI向け_業務フロー・ジャーニー記録テンプレート]]

### ① ヒアリング情報
```
ヒアリング日：YYYY-MM-DD
対象業務：〇〇業務
ヒアリング対象者：〇〇部 〇〇さん（役割：担当者/承認者）
```

> 💡 **コツ**：対象者の「役割（担当者/承認者/管理者）」を明記することで、視点の違いを後から把握しやすくなる。

---

### ② エンプロイージャーニー（現場のペインポイント）
```
🚨 現状のペイン
・（例：Excelファイルを毎日手作業で更新するのが手間）

ユーザーの生の声（Voice of User）
・「〜〜〜〜で本当に時間がかかる」
・「〜〜〜〜が自動化されたら助かる」

💡 期待する状態
・（例：申請フォームから1分で完了させたい）
```

> 💡 **コツ**：「生の声」は、できるだけそのまま（加工せず）書き残す。後でニュアンスが変わると要件にズレが生じる。

---

### ③ 業務フロー（テキストベース）
```
❌ 現状の業務フロー（As-Is）
1. [担当者] 〇〇をする
2. [担当者] 〇〇に提出する
   -> [分岐] 不備がある場合：〇〇が差し戻す → 1. に戻る

✅ 改善後の業務フロー（To-Be）
1. [担当者] システムから〇〇を入力・保存する
2. [システム] 自動で〇〇に承認依頼が飛ぶ
```

> 💡 **コツ**：`[担当者]` `[システム]` `[分岐]` を統一表記することで、誰が書いても読みやすくなる。

---

### ④ PBI要件（受入条件・実装メモ）
```
受入条件
- [ ] ユーザーが〇〇できること
- [ ] 保存時に〇〇が自動実行されること

必要な設定・開発要素（メモ）
- オブジェクト：〇〇__c の作成
- 自動化：〇〇フローの設定
```

---

## 実践のフロー

```
1. PBIのユーザーストーリーを読む
         ↓
2. 施策1の「ヒアリング設計シート」で事前準備
         ↓
3. ヒアリングを実施（30〜60分）
         ↓
4. 本テンプレートに記録する（当日中）
         ↓
5. チームに共有・フィードバックをもらう
```

---

## 運用のコツ

| ポイント | 内容 |
|--------|------|
| **タイミング** | ヒアリング当日中に記入する（翌日には記憶が薄れる） |
| **完璧を求めない** | 空欄があっても構わない。途中でも残すことが大事 |
| **チームに共有** | スプリントレビューやSlackでシェアして、フィードバックをもらう |
| **更新する** | 追加ヒアリングや仕様変更があれば随時更新する |
