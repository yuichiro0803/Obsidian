# 業務仕様書の課題とアプローチ

作成日：2026-03-01

---

## 課題・疑問点

### 背景

Salesforce 導入支援プロジェクトにおいて、**業務フロー図だけでは表現しきれない情報**があり、保守性の観点からドキュメントの不足を感じていた。

### 具体的な疑問点

> **「業務フロー図で表現しきれない、Salesforce上の動作も含めたドキュメントが必要ではないか」**

特に以下の情報を1つのドキュメントで整理したかった：

| 不足していた情報 | 内容 |
|----------------|------|
| **操作画面** | 各業務でユーザーがどの Salesforce 画面を操作するか |
| **自動処理の連鎖** | 保存時にどの Flow / Trigger / Approval が発火するか |
| **最終状態** | 処理完了後に各オブジェクト・項目がどの状態になるか |
| **エラーハンドリング** | エラー発生時にシステムがどう振る舞い、どう復旧するか |

### 課題の本質

業務フロー図は**人間の行動（誰が何をするか）**を表現するが、  
Salesforce システムの内部動作（自動処理の連鎖・データ状態変化）は表現できない。  
→ 開発・保守の現場では「画面とシステム動作を紐付けた仕様書」が必要。

---

## アプローチ

### 既存手法の調査結果

| 手法 | 概要 | 評価 |
|------|------|------|
| **ユースケース記述書（Fully Dressed）** | 事前条件・主フロー・代替フロー・例外フロー・事後条件を構造化して記述 | ⭐ 最も課題に近い概念 |
| **システムシーケンス図（SSD）** | ユーザー操作とシステム内部処理を時系列で可視化 | 外部システム連携の表現に特に有効 |
| **BPMN 2.0（拡張）** | Service Task / Error Event 等で自動処理・エラーを図で表現 | Salesforce 固有概念の表現に限界あり |
| **自動化マップ / 処理連鎖仕様書** | プロジェクト独自で作られることが多い実務的なドキュメント | カスタマイズ余地が高く現場に馴染みやすい |

### 結論：独自テンプレートの作成

上記手法を組み合わせ、**Salesforce 導入プロジェクト向けの統合テンプレート**を作成することで課題を解決する。

---

## 解決策：作成したテンプレート

📄 [[Salesforce業務・処理仕様書]]

### テンプレートの構成（9セクション）

```
1. 事前条件        ─ 操作前に満たすべきデータ状態・業務ルール
2. 操作画面        ─ ユーザーが操作する画面・導線
3. 主フロー（正常系）─ 操作〜自動処理の連鎖〜事後状態
4. 事後条件        ─ 処理後の各オブジェクト・項目の状態
5. 代替フロー      ─ 条件分岐・正常系以外のパターン
6. エラーハンドリング─ エラー条件・動作・表示・復旧手順
7. 自動処理マップ  ─ VR / Flow / Trigger / Approval の実行順一覧
8. 関連ドキュメント ─ 業務フロー図・ER図・テスト仕様書へのリンク
9. 変更履歴        ─ バージョン管理
```

### 運用ルール

- **1ファイル = 1ユースケース**（1業務シナリオ）でコピーして管理する
- ファイル命名規則：`UC-XXX_業務名.md`（例：`UC-001_商談クローズ処理.md`）
- 業務フロー図・ER図・テスト仕様書と**セットで管理**することで保守性が高まる

---

## 他手法との役割分担（まとめ）

```
業務の全体像（誰が何をするか）
  └─ 業務フロー図（スイムレーン）

Salesforce 上の操作〜自動処理〜状態変化
  └─ 【本テンプレート】Salesforce 業務・処理仕様書

外部システム連携の詳細
  └─ シーケンス図

ステータス管理・状態遷移ルール
  └─ ステートマシン図（状態遷移図）

データ構造・オブジェクト関連
  └─ ER 図 / ドメインモデル図
```
