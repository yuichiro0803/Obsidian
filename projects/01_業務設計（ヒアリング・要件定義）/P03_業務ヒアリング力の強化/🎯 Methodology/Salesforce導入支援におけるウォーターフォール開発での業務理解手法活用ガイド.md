# Salesforce 導入支援における業務理解手法 活用ガイド（ウォーターフォール）

Salesforce 導入支援プロジェクトにおいては、ウォーターフォール開発の各フェーズで「誰が・何を・いつ・どのように理解するか」を意識した手法選択が重要になる。  
本ドキュメントでは、各フェーズと業務理解手法の対応関係、活用観点・タイミング・ケースを整理する。

---

## ウォーターフォール開発の全体フェーズと手法マッピング

```
要件定義 → 基本設計 → 詳細設計 → 開発（設定・カスタマイズ）→ テスト → 移行・リリース → 運用保守
```

| フェーズ | 主な業務理解手法 |
|----------|---------------|
| 要件定義 | 業務フロー図、ユースケース図、ユーザーストーリーマップ、ジャーニーマップ（カスタマー/エンプロイー） |
| 基本設計 | 業務フロー図（To-Be）、イベントストーミング、ER図、ドメインモデル図 |
| 詳細設計 | シーケンス図、ステートマシン図（状態遷移図）、ER図 |
| 開発・設定 | ステートマシン図、シーケンス図 |
| テスト | 業務フロー図、ユースケース図（テストケース設計のベース） |
| 移行・リリース | 業務フロー図（移行フロー）、ユーザーストーリーマップ |

---

## フェーズ別 詳細活用ガイド

---

### 🔵 フェーズ1：要件定義

> **目的**：顧客の現状業務（As-Is）を正確に把握し、Salesforce 導入後の理想業務（To-Be）の方向性を合意する

---

#### ✅ 業務フロー図（スイムレーン）

| 項目 | 内容 |
|------|------|
| **観点** | 現状業務の「誰が・何を・どの順番でやるか」を可視化し、業務の全体像と担当者間の責任境界を明らかにする |
| **タイミング** | ヒアリング開始後すぐ。現状業務（As-Is）の棚卸しに使う |
| **活用ケース** | 営業フロー・案件管理プロセス・顧客対応フローなど、Salesforce に取り込む業務プロセスの現状把握 |
| **注意点** | 担当者によって認識がズレていることが多い。部署ごとに個別ヒアリングしてから統合する |

---

#### ✅ ユースケース図

| 項目 | 内容 |
|------|------|
| **観点** | 「誰が Salesforce の何を使うか」を一覧化する。ロール・権限設計の前提整理にもなる |
| **タイミング** | 業務フロー図の把握後、Salesforce の機能範囲（スコープ）を合意するタイミング |
| **活用ケース** | 営業担当・マネージャー・管理部門など複数ロールが Salesforce を使う場合。利用者ごとの機能スコープの整理 |
| **注意点** | 詳細なロジックは表現しない。あくまで「何ができるか」の概要整理に留める |

---

#### ✅ ユーザーストーリーマップ

| 項目 | 内容 |
|------|------|
| **観点** | ユーザーの行動シナリオ（横軸）をベースに、必要な機能を優先度・フェーズ（縦軸）で整理する |
| **タイミング** | スコープ合意後、Phase1／Phase2 など段階リリースの計画を立てるとき |
| **活用ケース** | 予算・期間が限られており、何を最初にリリースするかをステークホルダーと合意したいケース |
| **注意点** | ウォーターフォールであってもスコープ分割の議論に有効。アジャイル手法に限定されない |

---

#### ✅ ジャーニーマップ（カスタマー / エンプロイージャーニーマップ）

| 項目 | 内容 |
|------|------|
| **観点** | 従業員やユーザーの実際の業務体験・感情（不満点・要望）を可視化し、システム導入で「どこを最も改善すべきか」を発見する |
| **タイミング** | 要件定義初期。「なぜ Salesforce を入れるのか」「現場が何に困っているか」を整理するフェーズ |
| **活用ケース** | 現場の反発が予想される場合や、既存システムへの不満（ペインポイント）が大きい業務領域の洗い出し・改善提案 |
| **注意点** | 業務フロー図（手順の羅列）にならないよう、「感情の浮き沈み」や「現場の生の声（要望）」を中心に描く |

---

### 🟢 フェーズ2：基本設計

> **目的**：Salesforce 導入後の業務フロー（To-Be）を確定し、オブジェクト設計・データモデルの骨格を作る

---

#### ✅ 業務フロー図（To-Be）

| 項目 | 内容 |
|------|------|
| **観点** | Salesforce 導入後の「あるべき姿」の業務フローを可視化し、As-Is との差分・変更点を明確にする |
| **タイミング** | As-Is の業務フロー確定後、Salesforce の標準機能でどこまで対応できるかを検討しながら作成 |
| **活用ケース** | 業務変更が発生する場合（例：承認フローの変更・データ入力タイミングの変更）に、現行との差分を顧客と合意するとき |
| **注意点** | Salesforce の標準フローに合わせた業務変更を顧客に受け入れてもらう説得材料にもなる |

---

#### ✅ イベントストーミング

| 項目 | 内容 |
|------|------|
| **観点** | Salesforce 上で発生するドメインイベント（例：「商談がクローズされた」「ToDo が作成された」）を洗い出し、自動化・トリガーの設計に活かす |
| **タイミング** | 業務フロー（To-Be）の確定後、Salesforce の自動化（Flow / Apex Trigger）の設計を始めるとき |
| **活用ケース** | 複雑な承認フロー・自動通知・複数オブジェクト連動など、業務ルールが多いケース |
| **注意点** | Salesforce では「イベント」＝「レコードの状態変化」に対応するため、トリガー設計と直結させやすい |

---

#### ✅ ER 図 / ドメインモデル図

| 項目 | 内容 |
|------|------|
| **観点** | Salesforce の標準オブジェクト（Account, Contact, Opportunity など）とカスタムオブジェクトの関連を整理する |
| **タイミング** | 基本設計フェーズ。オブジェクト設計・カスタムオブジェクト作成の前に作成する |
| **活用ケース** | 標準オブジェクトだけでは業務要件を満たせず、カスタムオブジェクト追加や外部システム連携が発生するケース |
| **注意点** | Salesforce は「オブジェクト＝テーブル」ではなく関連定義（Lookup / Master-Detail）が重要なため、関連の方向・多重度を明確に |

---

### 🟡 フェーズ3：詳細設計

> **目的**：Salesforce の画面設計・自動化処理・連携仕様を具体的に定義する

---

#### ✅ シーケンス図

| 項目 | 内容 |
|------|------|
| **観点** | Salesforce と外部システム（ERP・MA・Slackなど）の連携フローを時系列で可視化する |
| **タイミング** | 外部システム連携（API 連携・データ同期）の詳細設計時 |
| **活用ケース** | Salesforce ↔ 基幹システム連携・Salesforce → Slack 通知・外部 API 呼び出しなど、複数システムをまたぐ処理の設計 |
| **注意点** | 非同期処理（Platform Events / Queueable Apex）は同期シーケンスと分けて描く |

---

#### ✅ ステートマシン図（状態遷移図）

| 項目 | 内容 |
|------|------|
| **観点** | Salesforce のレコードステータス（例：商談フェーズ・ケースステータス）の遷移ルール・条件を明確にする |
| **タイミング** | 承認プロセス・レコードタイプ・フェーズ管理などを設計するとき |
| **活用ケース** | 商談フェーズの進捗管理・ケース（サポートチケット）のステータス管理・注文ステータス管理など「ステータスが複雑なオブジェクト」の設計 |
| **注意点** | Salesforce の Validation Rule・Flow で「どの状態からどの状態に遷移できるか」の制御と対応させる |

---

### 🔴 フェーズ4：開発（設定・カスタマイズ）

> **目的**：Salesforce の各種設定・カスタム開発（Apex / Flow / LWC）を実装する

---

#### ✅ シーケンス図（参照用）

| 項目 | 内容 |
|------|------|
| **観点** | 詳細設計で作成したシーケンス図を開発の仕様書として参照する |
| **タイミング** | Apex クラス・Trigger・Flow の実装時 |
| **活用ケース** | 開発者がシーケンス図を見ながら実装する。コードレビュー時の設計意図確認にも使う |

---

#### ✅ ステートマシン図（参照用）

| 項目 | 内容 |
|------|------|
| **観点** | 状態遷移ルールを Validation Rule・Flow・Apex に落とし込む際の仕様確認 |
| **タイミング** | ステータス管理の実装時・テストデータ設計時 |
| **活用ケース** | レコードタイプ・承認プロセスの設定時に、遷移ルールが漏れていないか確認する |

---

### 🟣 フェーズ5：テスト

> **目的**：Salesforce の動作が業務要件・設計仕様を満たしているかを検証する

---

#### ✅ 業務フロー図（テストシナリオ設計）

| 項目 | 内容 |
|------|------|
| **観点** | To-Be 業務フロー図をベースに、テストシナリオ（シナリオテスト）を設計する |
| **タイミング** | システムテスト・ユーザー受入テスト（UAT）の設計フェーズ |
| **活用ケース** | 業務担当者が UAT を実施する際に、フロー図をテスト手順書の骨格として活用する |
| **注意点** | 分岐条件・例外ケース（エラーパターン）もフロー図上で明示し、テストケースの抜け漏れを防ぐ |

---

#### ✅ ユースケース図（テストカバレッジ確認）

| 項目 | 内容 |
|------|------|
| **観点** | 定義したユースケースがすべてテストされているかをチェックリストとして活用する |
| **タイミング** | テスト計画・テストケースレビュー時 |
| **活用ケース** | 複数ロール（営業・マネージャー・管理部）が異なる操作をするシステムで、ロールごとのテスト漏れを防ぐ |

---

### ⚪ フェーズ6：移行・リリース

> **目的**：データ移行・ユーザートレーニング・本番リリースを安全に実施する

---

#### ✅ 業務フロー図（移行・展開フロー）

| 項目 | 内容 |
|------|------|
| **観点** | データ移行作業・リリース手順・運用切り替えのフローを可視化し、関係者間で手順を共有する |
| **タイミング** | リリース計画策定時・リリース直前のリハーサル時 |
| **活用ケース** | データ移行（旧システム → Salesforce）の手順・ロールバック計画の共有。複数部署が関わるリリース作業の調整 |

---

#### ✅ ユーザーストーリーマップ（トレーニング計画）

| 項目 | 内容 |
|------|------|
| **観点** | ユーザーの行動シナリオを元にトレーニング内容・マニュアル構成を決める |
| **タイミング** | ユーザートレーニング実施前 |
| **活用ケース** | エンドユーザー向けマニュアル・ハンズオン研修の構成を、ユーザーの実際の業務フローに沿って設計したいケース |

---

## 手法選択のポイント（Salesforce 導入特有の観点）

### 1. 標準機能 vs カスタマイズの見極めに「業務フロー図 + イベントストーミング」を使う

- 業務フロー（As-Is）と Salesforce 標準フローを比較し、**GAP 分析**に使う
- GAP が大きい業務領域にはカスタマイズが必要 → イベントストーミングで自動化設計

### 2. オブジェクト設計は「ER 図 + ドメインモデル図」で両面から整理する

- ER 図：データの持ち方・関連の多重度を整理
- ドメインモデル図：業務上の「意味・ルール」を整理（例：「Account ＝ 取引先 ＝ 法人単位」）

### 3. 承認フロー・複雑なステータス管理は「ステートマシン図」で先に整理する

- Salesforce の Approval Process / Flow の設計前に状態遷移を文書化しておくと、設定ミスを防げる

### 4. 外部システム連携（API）は「シーケンス図」で必ず可視化する

- Salesforce ↔ 外部システムのリクエスト/レスポンス・エラーハンドリングを明確に
- Connected App・Named Credentials の設計の根拠にもなる

### 5. UAT（ユーザー受入テスト）は「To-Be 業務フロー図」をそのままシナリオ台本にする

- 業務担当者が理解しやすい言葉（IT 用語を使わない）でフローを記述しておくと UAT の円滑化につながる
